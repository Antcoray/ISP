## 3.3.3 Доменная модель и архитектура

### 1. Что такое «Чистая» архитектура? Её преимущества перед трёхслойной.

**Чистая архитектура (Clean Architecture)** — архитектурный подход, в котором приложение делится на концентрические слои с жёстким правилом: зависимости направлены **только внутрь** (к ядру). Внутренние слои ничего не знают о внешних.

Слои в данном проекте:

| Слой | Проект | Зависит от |
|---|---|---|
| Domain | `Kolesnikow_453501.Domain` | ни от кого |
| Application | `Kolesnikow_453501.Application` | Domain |
| Persistence | `Kolesnikow_453501.Persistence` | Domain |
| UI | `Kolesnikow_453501.UI` | Application, Persistence, Domain |

**Преимущества перед трёхслойной архитектурой:**
- Доменная логика полностью изолирована от технологий (БД, UI, фреймворков).
- Легко подменить реализацию инфраструктуры (например, `FakeUnitOfWork` → `EfUnitOfWork`) без изменения бизнес-логики.
- Независимая тестируемость каждого слоя.
- В трёхслойной архитектуре UI или DAL может напрямую зависеть от бизнес-логики и обратно — в Clean Architecture это запрещено.

**В проекте:** `Contest` и `Participant` в `Domain` не зависят ни от EF Core, ни от MAUI. Слой `Persistence` реализует интерфейсы из `Domain`, не наоборот.

---

### 2. Что такое «Принцип устойчивых зависимостей» (SDP)?

**SDP (Stable Dependencies Principle)** — компонент должен зависеть только от компонентов, которые более устойчивы, чем он сам.

Устойчивый компонент — тот, от которого зависят многие другие, и который сам ни от кого не зависит. Нестабильный — тот, на который мало кто полагается, и который зависит от многих.

**В проекте:** `Domain` — самый устойчивый слой (от него зависят Application, Persistence и UI, а сам он ни от кого). `UI` — самый нестабильный (зависит от всех остальных). Это соответствует SDP.

---

### 3. Как оцениваются метрики устойчивости?

Метрика **нестабильности (I)**:

```
I = Fan-out / (Fan-in + Fan-out)
```

- **Fan-in** — количество входящих зависимостей (кто зависит от данного компонента).
- **Fan-out** — количество исходящих зависимостей (от кого зависит данный компонент).
- `I = 0` — абсолютно устойчивый, `I = 1` — абсолютно нестабильный.

**В проекте:**
- `Domain`: Fan-in = 3, Fan-out = 0 → I = 0 (максимально устойчив).
- `UI`: Fan-in ≈ 0, Fan-out = 3 → I ≈ 1 (максимально нестабилен).

### 4. Что такое UnitOfWork?

**Unit of Work (единица работы)** — шаблон проектирования, который отслеживает все изменения объектов в рамках одной транзакции и фиксирует их одним вызовом `SaveAll`.

Предоставляет единую точку доступа к репозиториям и гарантирует атомарность операций: либо все изменения сохраняются, либо ни одно.

**В проекте:**
- Интерфейс: [`IUnitOfWork.cs`](Kolesnikow_453501.Domain/Abstractions/IUnitOfWork.cs)
- EF-реализация: [`EfUnitOfWork.cs`](Kolesnikow_453501.Persistence/Repository/EfUnitOfWork.cs) — делегирует `SaveAllAsync()` → `_context.SaveChangesAsync()`.
- Fake-реализация: [`FakeRepositories.cs`](Kolesnikow_453501.Persistence/Repository/FakeRepositories.cs) — для разработки без БД.

```csharp
public interface IUnitOfWork
{
    IRepository<Contest> ContestRepository { get; }
    IRepository<Participant> ParticipantRepository { get; }
    Task SaveAllAsync();
    Task DeleteDataBaseAsync();
    Task CreateDataBaseAsync();
}
```

### 5. В каком слое чистой архитектуры находится контекст базы данных?

В слое **Persistence** (инфраструктурный слой). Контекст БД — деталь реализации, о которой Domain и Application ничего не знают.

**В проекте:** [`AppDbContext.cs`](Kolesnikow_453501.Persistence/Data/AppDbContext.cs) объявлен как `internal sealed` — он недоступен за пределами сборки `Persistence`. Слой UI передаёт только строку подключения, не зная о существовании `AppDbContext`.

### 6. Шаблоны «Фабричный метод» и «Абстрактная фабрика»

**Фабричный метод (Factory Method)** — определяет интерфейс для создания объекта, но позволяет подклассам решать, какой класс создавать. Создание вынесено в отдельный метод.

```csharp
// Пример: метод, возвращающий IRepository<T>
IRepository<Contest> CreateRepository() => new EfRepository<Contest>(context);
```

**Абстрактная фабрика (Abstract Factory)** — предоставляет интерфейс для создания *семейств* взаимосвязанных объектов без указания конкретных классов.

```csharp
// Пример: IUnitOfWork как абстрактная фабрика репозиториев
IRepository<Contest> ContestRepository { get; }
IRepository<Participant> ParticipantRepository { get; }
```

**В проекте:** `IUnitOfWork` играет роль абстрактной фабрики — предоставляет доступ к семейству репозиториев. `EfUnitOfWork` — конкретная фабрика для EF Core, `FakeUnitOfWork` — для тестирования.

---

### 7. Шаблон проектирования «Адаптер»

**Адаптер (Adapter)** — структурный шаблон, преобразующий интерфейс одного класса в интерфейс, ожидаемый клиентом. Позволяет совместно использовать классы с несовместимыми интерфейсами.

**В проекте:** `EfRepository<T>` адаптирует `DbSet<T>` (интерфейс EF Core) к `IRepository<T>` (интерфейс Domain). Domain не знает о EF Core — он работает только с `IRepository<T>`, а `EfRepository<T>` «переводит» вызовы:

```csharp
// IRepository<T>.ListAsync(...) → DbSet<T>.AsQueryable().Where(...).ToListAsync()
```

---

### 8. Какой механизм используется для изменения направления зависимости?

**Инверсия зависимостей (Dependency Inversion)** через интерфейсы. Вместо того чтобы Application зависел от Persistence напрямую, оба зависят от абстракции в Domain.

Конкретно: **Dependency Injection (внедрение зависимостей)** — объекты получают зависимости извне через конструктор, а не создают их сами. Контейнер IoC (.NET `IServiceCollection`) сопоставляет интерфейс с реализацией.

**В проекте:**
- [`DependencyInjection.cs (Persistence)`](Kolesnikow_453501.Persistence/DependencyInjection.cs): `services.AddSingleton<IUnitOfWork, EfUnitOfWork>()`
- [`MauiProgram.cs`](Kolesnikow_453501.UI/MauiProgram.cs): `builder.Services.AddApplication().AddPersistence(connStr!)`
- Обработчики MediatR получают `IUnitOfWork` через конструктор — они не знают, `EfUnitOfWork` это или `FakeUnitOfWork`.

---

### 9. Что означает «S» в аббревиатуре SOLID?

**S — Single Responsibility Principle (принцип единственной ответственности).**

Класс должен иметь только одну причину для изменения, то есть отвечать только за одну задачу.

**В проекте:**
- `Contest` отвечает только за доменную логику номинации.
- `EfRepository<T>` — только за доступ к данным через EF Core.
- `ContestsViewModel` — только за подготовку данных для `ContestsPage`.
- `AddContestCommandHandler` — только за обработку команды добавления номинации.

Каждый класс имеет одну зону ответственности, что соответствует SRP.

---

### 10. REP: Reuse/Release Equivalence Principle

**Принцип эквивалентности повторного использования и выпусков** — единица повторного использования равна единице выпуска. Нельзя повторно использовать компонент без механизма управления его версиями (релизами).

Практический смысл: если вы публикуете библиотеку, которую используют другие, необходимо версионировать её как единое целое. Всё, что входит в один компонент (сборку), должно быть логически связано и выпускаться вместе.

**В проекте:** каждый проект решения (`Domain`, `Application`, `Persistence`, `UI`) является отдельной единицей выпуска (сборкой). Если изменяется `IRepository<T>` в `Domain`, требуется новый выпуск `Domain`, и все зависящие сборки пересобираются.

## 3.4.6 Слой Persistence и Entity Framework Core

### Что такое EntityFramework Core?

**Entity Framework Core (EF Core)** — легковесный, расширяемый ORM-фреймворк от Microsoft для .NET. Позволяет работать с базой данных через объекты C#, автоматически генерируя SQL-запросы.

**В проекте:** EF Core используется в `Kolesnikow_453501.Persistence`. Контекст [`AppDbContext`](Kolesnikow_453501.Persistence/Data/AppDbContext.cs) наследует `DbContext`. [`EfRepository<T>`](Kolesnikow_453501.Persistence/Repository/EfRepository.cs) выполняет CRUD-операции через `DbSet<T>`.

---

### Как описать контекст БД в EntityFramework Core?

Создаётся класс, наследующий `DbContext`. Конструктор принимает `DbContextOptions<TContext>`. В методе `OnModelCreating` настраиваются связи и маппинг.

**В проекте** ([`AppDbContext.cs`](Kolesnikow_453501.Persistence/Data/AppDbContext.cs)):

```csharp
internal sealed class AppDbContext : DbContext
{
    public DbSet<Contest> Contests { get; set; }
    public DbSet<Participant> Participants { get; set; }

    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options)
    {
        Database.EnsureCreated();
    }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Contest>()
            .HasMany(c => c.ParticipantsList)
            .WithOne(p => p.Contest)
            .HasForeignKey(p => p.ContestId);
        modelBuilder.Entity<Contest>()
            .Ignore(c => c.Participants);
    }
}
```

---

### Как указать базу данных в EF Core?

Через метод расширения провайдера в `DbContextOptionsBuilder`. Для SQLite — `.UseSqlite(connectionString)`.

**В проекте** ([`DependencyInjection.cs`](Kolesnikow_453501.Persistence/DependencyInjection.cs)):

```csharp
var options = new DbContextOptionsBuilder<AppDbContext>()
    .UseSqlite(connectionString)
    .Options;
```

Строка подключения берётся из [`appsettings.json`](Kolesnikow_453501.UI/appsettings.json): `"Data Source = {0}Kolesnikow.db"`, где `{0}` — путь к директории данных приложения.

---

### Как указать отношение 1-ко-многим в EF Core?

Через Fluent API в `OnModelCreating`:

```csharp
modelBuilder.Entity<Contest>()
    .HasMany(c => c.ParticipantsList)   // одна номинация имеет много участников
    .WithOne(p => p.Contest)             // каждый участник принадлежит одной номинации
    .HasForeignKey(p => p.ContestId);    // внешний ключ
```

**В проекте:** применено в [`AppDbContext.cs`](Kolesnikow_453501.Persistence/Data/AppDbContext.cs). Таблица `Participants` содержит колонку `ContestId` как внешний ключ на `Contests`.

---

### Что такое шаблон проектирования «Построитель» (Builder)?

**Builder** — порождающий шаблон, который разделяет конструирование сложного объекта от его представления, позволяя создавать разные конфигурации одного объекта пошагово.

**В проекте:** `DbContextOptionsBuilder<AppDbContext>` — классический пример Builder:

```csharp
var options = new DbContextOptionsBuilder<AppDbContext>()
    .UseSqlite(connectionString)   // шаг 1: указать провайдер
    .Options;                      // получить готовый объект
```

Также `MauiApp.CreateBuilder()` в [`MauiProgram.cs`](Kolesnikow_453501.UI/MauiProgram.cs) строит приложение пошагово через цепочку вызовов.

---

### Для чего используются миграции EF Core?

**Миграции** — механизм EF Core для управления схемой БД: каждое изменение модели данных фиксируется как миграция (C#-файл с методами `Up` и `Down`), которую можно применить или откатить.

**В данном проекте миграции не используются** — вместо них применяется `Database.EnsureCreated()` (создаёт БД по текущей модели если её нет) и `Database.EnsureDeletedAsync()` + `EnsureCreatedAsync()` в [`DbInitializer.cs`](Kolesnikow_453501.Application/DbInitializer.cs) для пересоздания при старте. Это упрощает разработку, но не подходит для продакшена.

---

### Чем отличается `Expression<Func<T, bool>>` от `Func<T, bool>`?

| | `Func<T, bool>` | `Expression<Func<T, bool>>` |
|---|---|---|
| Что это | Делегат (скомпилированный код) | Дерево выражений (структура данных) |
| Выполнение | В памяти (LINQ to Objects) | Транслируется в SQL (LINQ to Entities) |
| Применение | `IEnumerable<T>` | `IQueryable<T>` |

**В проекте** [`EfRepository.cs`](Kolesnikow_453501.Persistence/Repository/EfRepository.cs) использует `Expression<Func<T, bool>>` — EF Core транслирует выражение в SQL `WHERE`, фильтрация происходит на стороне БД. `FakeParticipantRepository` использует `AsQueryable().Where(filter)` — работает с `Expression` в памяти.

## 3.5.3 Слой Application: CRUD и CQRS

### Что такое операции CRUD?

**CRUD** — базовые операции над данными:
- **C**reate — создание
- **R**ead — чтение
- **U**pdate — обновление
- **D**elete — удаление

**В проекте** реализованы через MediatR-команды и запросы:
- Create: [`AddContestCommand`](Kolesnikow_453501.Application/ContestUseCases/Commands/ContestCommands.cs), `AddParticipantCommand`
- Read: [`GetAllContestsRequest`](Kolesnikow_453501.Application/ContestUseCases/Queries/ContestQueries.cs), `GetParticipantsByContestRequest`
- Update: `UpdateContestCommand`, `UpdateParticipantCommand`
- Delete: `DeleteContestCommand`, `DeleteParticipantCommand`

---

### Что такое CQRS?

**CQRS (Command Query Responsibility Segregation)** — разделение ответственности между командами и запросами:
- **Query (запрос)** — только читает данные, не изменяет состояние системы.
- **Command (команда)** — изменяет состояние, не возвращает данные (или возвращает минимум).

Это позволяет независимо масштабировать и оптимизировать операции чтения и записи.

**В проекте** реализован через библиотеку **MediatR**:
- Папки `ContestUseCases/Queries` и `ParticipantUseCases/Queries` — запросы (только чтение).
- Папки `ContestUseCases/Commands` и `ParticipantUseCases/Commands` — команды (изменение данных).

```csharp
// Запрос — только читает
public sealed record GetAllContestsRequest() : IRequest<IEnumerable<Contest>>;

// Команда — изменяет данные
public sealed record AddContestCommand(string Name, DateTime Date, string Description)
    : IRequest<Contest>;
```

## 3.6.2 Подготовка проекта UI

### Что такое шаблон проектирования MVVM?

**MVVM (Model — View — ViewModel)** — архитектурный шаблон для UI-приложений:
- **Model** — бизнес-сущности и логика (классы `Contest`, `Participant`).
- **View** — разметка UI (XAML-страницы, не содержит логики).
- **ViewModel** — посредник: предоставляет View данные и команды, реагирует на действия пользователя.

View привязана к ViewModel через **Data Binding** — изменения в ViewModel автоматически отражаются в UI.

**В проекте:**
- View: [`ContestsPage.xaml`](Kolesnikow_453501.UI/Pages/ContestsPage.xaml), [`ParticipantDetailsPage.xaml`](Kolesnikow_453501.UI/Pages/ParticipantDetailsPage.xaml)
- ViewModel: [`ContestsViewModel.cs`](Kolesnikow_453501.UI/ViewModels/ContestsViewModel.cs), [`ParticipantDetailsViewModel.cs`](Kolesnikow_453501.UI/ViewModels/ParticipantDetailsViewModel.cs)
- Model: [`Contest.cs`](Kolesnikow_453501.Domain/Entities/Contest.cs), [`Participant.cs`](Kolesnikow_453501.Domain/Entities/Participant.cs)

---

### Как настроить контейнер IoC в .NET MAUI?

В методе `CreateMauiApp()` файла `MauiProgram.cs` через `builder.Services` (стандартный `IServiceCollection` из Microsoft.Extensions.DependencyInjection):

```csharp
builder.Services
    .AddApplication()        // регистрация MediatR-обработчиков
    .AddPersistence(connStr) // регистрация IUnitOfWork → EfUnitOfWork
    .RegisterPages()         // регистрация страниц
    .RegisterViewModels();   // регистрация ViewModel
```

**В проекте** ([`MauiProgram.cs`](Kolesnikow_453501.UI/MauiProgram.cs)): все сервисы регистрируются до вызова `builder.Build()`. Страницы получают ViewModel через конструктор автоматически.

---

### Чем отличается сервис «singleton» от «transient»?

| | Singleton | Transient |
|---|---|---|
| Создаётся | Один раз на всё приложение | При каждом запросе из контейнера |
| Время жизни | Всё время работы приложения | Только пока используется |
| Применение | Общее состояние, тяжёлые ресурсы | Лёгкие объекты без состояния |

**В проекте** ([`MauiProgram.cs`](Kolesnikow_453501.UI/MauiProgram.cs)):
- `ContestsPage` и `ContestsViewModel` — `AddSingleton`: страница одна, хранит список номинаций.
- `ParticipantDetailsPage`, `AddEditContestPage`, их ViewModel — `AddTransient`: создаются заново при каждом переходе, не хранят устаревшие данные.
- `IUnitOfWork` (`EfUnitOfWork`) — `AddSingleton`: один `DbContext` на приложение.

## 3.7.7 MVVM: привязки, команды, конвертеры

### Для чего используется `INotifyPropertyChanged`?

Интерфейс уведомляет UI о том, что значение свойства изменилось. Без него привязка (`Binding`) не обновит элемент управления при изменении свойства ViewModel.

```csharp
public class MyViewModel : INotifyPropertyChanged
{
    public event PropertyChangedEventHandler? PropertyChanged;

    private string _name = "";
    public string Name
    {
        get => _name;
        set { _name = value; PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(nameof(Name))); }
    }
}
```

**В проекте:** реализация делегируется `ObservableObject` из `CommunityToolkit.Mvvm` — атрибут `[ObservableProperty]` генерирует свойство с вызовом `OnPropertyChanged()` автоматически. Пример: `[ObservableProperty] Contest? selectedContest` в [`ContestsViewModel.cs`](Kolesnikow_453501.UI/ViewModels/ContestsViewModel.cs).

---

### Для чего используется `ObservableCollection<T>`?

Коллекция, которая уведомляет UI (через событие `CollectionChanged`) о добавлении, удалении или замене элементов. Обычный `List<T>` не уведомляет — `CollectionView` не обновится при изменении.

**В проекте** ([`ContestsViewModel.cs`](Kolesnikow_453501.UI/ViewModels/ContestsViewModel.cs)):

```csharp
public ObservableCollection<Contest> Contests { get; set; } = new();
public ObservableCollection<Participant> Participants { get; set; } = new();
```

При вызове `Contests.Clear()` и `Contests.Add(...)` `Picker` на странице обновляется автоматически.

---

### Как привязать значение XAML к свойству ViewModel?

Через синтаксис `{Binding PropertyName}`. ViewModel должна быть назначена как `BindingContext` страницы.

**В проекте** ([`ContestsPage.xaml`](Kolesnikow_453501.UI/Pages/ContestsPage.xaml)):

```xml
<!-- Страница знает тип ViewModel для статической проверки -->
x:DataType="models:ContestsViewModel"

<!-- Привязка выбранного элемента Picker к свойству SelectedContest -->
<Picker ItemsSource="{Binding Contests}"
        SelectedItem="{Binding SelectedContest}"
        ItemDisplayBinding="{Binding Name}" />
```

BindingContext назначается в code-behind ([`ContestsPage.xaml.cs`](Kolesnikow_453501.UI/Pages/ContestsPage.xaml.cs)):
```csharp
BindingContext = viewModel;
```

---

### Для чего используется `ICommand`?

Интерфейс позволяет привязать действие (метод) ViewModel к элементу UI (кнопке, жесту и т.д.) через Data Binding, не создавая обработчиков событий в code-behind.

```csharp
public interface ICommand
{
    bool CanExecute(object? parameter);
    void Execute(object? parameter);
    event EventHandler? CanExecuteChanged;
}
```

**В проекте:** реализация генерируется атрибутом `[RelayCommand]` (CommunityToolkit.Mvvm). Пример из [`ContestsViewModel.cs`](Kolesnikow_453501.UI/ViewModels/ContestsViewModel.cs):

```csharp
[RelayCommand]
async Task UpdateGroupList() => await GetContests();
// Генерирует свойство UpdateGroupListCommand типа IAsyncRelayCommand
```

---

### Что такое конвертер значений (Value Converter)?

Класс, реализующий `IValueConverter`, преобразует значение из одного типа в другой при привязке XAML → UI. Позволяет, например, преобразовать число в цвет.

**В проекте** ([`VotesToColorValueConverter.cs`](Kolesnikow_453501.UI/ValueConverters/VotesToColorValueConverter.cs)):

```csharp
// Votes < 50 → красная рамка, иначе прозрачная
if ((int)value < 50) return Colors.Red;
return Colors.Transparent;
```

Применяется в [`ContestsPage.xaml`](Kolesnikow_453501.UI/Pages/ContestsPage.xaml): рамка карточки участника меняет цвет в зависимости от количества баллов.

Второй конвертер: [`IdToImageSourceValueConverter.cs`](Kolesnikow_453501.UI/ValueConverters/IdToImageSourceValueConverter.cs) — преобразует `Id` участника в `ImageSource` из файла `Images/{id}.jpg`.

### Что такое Binding Context?

`BindingContext` — объект, который является источником данных для всех привязок на странице или элементе управления. Когда `{Binding Name}` встречается в XAML, имя ищется в `BindingContext`.

**В проекте:** каждая страница устанавливает свой ViewModel как `BindingContext` в конструкторе:

```csharp
// ContestsPage.xaml.cs
BindingContext = viewModel; // viewModel — ContestsViewModel
```

### Как связать событие с командой?

Через `EventToCommandBehavior` из `CommunityToolkit.Maui` — позволяет привязать любое событие элемента управления к команде ViewModel.

**В проекте** ([`ContestsPage.xaml`](Kolesnikow_453501.UI/Pages/ContestsPage.xaml)):

```xml
<!-- Загрузка списка номинаций при загрузке страницы -->
<ContentPage.Behaviors>
    <toolkit:EventToCommandBehavior
        EventName="Loaded"
        Command="{Binding UpdateGroupListCommand}" />
</ContentPage.Behaviors>

<!-- Загрузка участников при выборе номинации в Picker -->
<Picker.Behaviors>
    <toolkit:EventToCommandBehavior
        EventName="SelectedIndexChanged"
        Command="{Binding UpdateMembersListCommand}" />
</Picker.Behaviors>
```

## 3.8.5 Работа с базой данных

### Как получить значение секции из `appsettings.json`?

Через `IConfiguration`, который регистрируется в `builder.Configuration`:

```csharp
// Получение строки подключения из секции ConnectionStrings
var connStr = builder.Configuration.GetConnectionString("SqliteConnection");

// Получение произвольного значения
var value = builder.Configuration["SomeSection:SomeKey"];
```

**В проекте** ([`MauiProgram.cs`](Kolesnikow_453501.UI/MauiProgram.cs)): файл `appsettings.json` подключается как `EmbeddedResource` и читается через `GetManifestResourceStream`. Строка подключения содержит placeholder `{0}`, который заменяется на путь к папке данных приложения.

---

### Для чего используются миграции в EF Core?

Миграции позволяют **эволюционно изменять схему БД** без потери данных. Каждое изменение модели (новое поле, таблица, связь) создаёт файл миграции с методами `Up()` (применить) и `Down()` (откатить).

Без миграций при изменении модели нужно либо пересоздавать БД (потеря данных), либо писать SQL вручную.

**В проекте:** миграции не применяются — используется `EnsureCreated()`/`EnsureDeleted()` с повторной инициализацией. Это приемлемо для учебного проекта, но в продакшене нужны миграции.

---

### Где находится контекст БД в трёхслойной архитектуре?

В **слое доступа к данным (DAL, Data Access Layer)** — аналог слоя Persistence в Clean Architecture.

В трёхслойной архитектуре: Presentation → Business Logic → Data Access. `DbContext` находится в Data Access, но часто туда «просачивается» бизнес-логика. В Clean Architecture это строго разделено — `AppDbContext` инкапсулирован в `Persistence` и недоступен другим слоям.

---

### Где по умолчанию сохраняются файлы приложения?

| Платформа | Путь |
|---|---|
| Windows | `%LocalAppData%\Packages\<PackageName>\LocalState\` |
| Android | `/data/data/<package>/files/` |
| iOS | `<AppBundle>/Library/` |

Получить путь в коде: `FileSystem.Current.AppDataDirectory`.

**В проекте** ([`MauiProgram.cs`](Kolesnikow_453501.UI/MauiProgram.cs)):
```csharp
string dataDirectory = FileSystem.Current.AppDataDirectory + "/";
connStr = string.Format(connStr!, dataDirectory);
// → "Data Source = C:\Users\...\LocalState/Kolesnikow.db"
```

Фото участников сохраняются в `AppDataDirectory/Images/{id}.jpg` ([`ParticipantDetailsViewModel.cs`](Kolesnikow_453501.UI/ViewModels/ParticipantDetailsViewModel.cs)).

---

### Как используется `IPreferences` / `Preferences`?

`Preferences` — платформонезависимое хранилище пар ключ-значение (аналог `SharedPreferences` на Android / `NSUserDefaults` на iOS / реестра на Windows). Подходит для хранения небольших настроек.

```csharp
// Запись
Preferences.Default.Set("theme", "dark");

// Чтение (с дефолтным значением)
string theme = Preferences.Default.Get("theme", "light");

// Удаление
Preferences.Default.Remove("theme");
```

В данном проекте `Preferences` не используется — хранение данных реализовано через SQLite.

## 3.9.4 Навигация и страница деталей

### Что такое «Словарь ресурсов» (Resource Dictionary)?

Коллекция ресурсов XAML (стили, конвертеры, шаблоны, цвета), доступных через ключ `{StaticResource Key}` или `{DynamicResource Key}`. Позволяет переиспользовать оформление без дублирования.

```xml
<ContentPage.Resources>
    <converters:VotesToColorValueConverter x:Key="VotesToColor" />
    <Style x:Key="CardStyle" TargetType="Border">
        <Setter Property="CornerRadius" Value="10" />
    </Style>
</ContentPage.Resources>
```

**В проекте** ([`ContestsPage.xaml`](Kolesnikow_453501.UI/Pages/ContestsPage.xaml), [`ParticipantDetailsPage.xaml`](Kolesnikow_453501.UI/Pages/ParticipantDetailsPage.xaml)): конвертеры `VotesToColorValueConverter` и `IdToImageSourceValueConverter` зарегистрированы в ресурсах страниц.

---

### Как использовать ресурс стиля в разметке XAML?

```xml
<!-- Определение -->
<ContentPage.Resources>
    <Style x:Key="TitleLabel" TargetType="Label">
        <Setter Property="FontAttributes" Value="Bold" />
        <Setter Property="FontSize" Value="18" />
    </Style>
</ContentPage.Resources>

<!-- Использование -->
<Label Text="Номинации" Style="{StaticResource TitleLabel}" />
```

`StaticResource` разрешается один раз при загрузке; `DynamicResource` реагирует на изменения ресурса во время выполнения.

---

### Для чего используется `CollectionView`? Как настроить внешний вид?

`CollectionView` — элемент управления для отображения списков с высокой производительностью (виртуализация). Заменяет `ListView`.

Настройка через `ItemTemplate`:

```xml
<CollectionView ItemsSource="{Binding Participants}">
    <CollectionView.ItemTemplate>
        <DataTemplate x:DataType="entities:Participant">
            <Border Stroke="{Binding Votes, Converter={StaticResource VotesToColor}}"
                    StrokeShape="RoundRectangle 8">
                <Label Text="{Binding Name}" />
            </Border>
        </DataTemplate>
    </CollectionView.ItemTemplate>
</CollectionView>
```

**В проекте** ([`ContestsPage.xaml`](Kolesnikow_453501.UI/Pages/ContestsPage.xaml)): `CollectionView` отображает список участников. Цвет рамки определяется `VotesToColorValueConverter` по полю `Votes`. При нажатии на участника `TapGestureRecognizer` вызывает `ShowDetailsCommand`.

### Как перейти на другую страницу из ViewModel?

Через `Shell.Current.GoToAsync()`:

### Как передать параметры при навигации?

Через словарь параметров в `GoToAsync` и атрибут `[QueryProperty]` на принимающей ViewModel:

```csharp
// Отправитель (ContestsViewModel)
var param = new Dictionary<string, object> { { "Participant", selectedParticipant } };
await Shell.Current.GoToAsync(nameof(ParticipantDetailsPage), param);

// Получатель (ParticipantDetailsViewModel)
[QueryProperty(nameof(Participant), "Participant")]
public partial class ParticipantDetailsViewModel : ObservableObject
{
    [ObservableProperty]
    Participant? participant;
}
```

**В проекте:** применяется в [`ParticipantDetailsViewModel.cs`](Kolesnikow_453501.UI/ViewModels/ParticipantDetailsViewModel.cs), [`AddEditContestViewModel.cs`](Kolesnikow_453501.UI/ViewModels/AddEditContestViewModel.cs), [`AddEditParticipantViewModel.cs`](Kolesnikow_453501.UI/ViewModels/AddEditParticipantViewModel.cs).

### Чем отличается переход по адресу `"PageName"` от `"//PageName"`?

| Адрес | Поведение |
|---|---|
| `"PageName"` | Push: добавляет страницу на стек. Можно вернуться кнопкой «Назад». |
| `"//PageName"` | Абсолютный маршрут: сбрасывает весь стек навигации и переходит к корневой странице. Кнопки «Назад» не будет. |

`".."` — переход назад (pop со стека).  
`"../../"` — переход на два уровня назад.

**В проекте:** все переходы используют относительный адрес (`nameof(...)`) — добавляют страницу на стек. Возврат выполняется через `GoToAsync("..")`.
